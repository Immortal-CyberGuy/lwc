Create src/crypto/speck_engine.py:import struct, hmac, hashlib, osfrom .base_engine import CryptoEngineclass SpeckCipher:    '''SPECK 128/128: 128-bit block, 128-bit key, 32 rounds'''    ROUNDS = 32    WORD_SIZE = 64    MOD = 2 ** 64    def __init__(self, key_bytes):        assert len(key_bytes) == 16        # Parse key as two 64-bit words (big-endian)        self.key_schedule = self._expand_key(key_bytes)    def _expand_key(self, key_bytes):        b, a = struct.unpack('>QQ', key_bytes)  # l[0], k[0]        k = [a]        l = [b]        for i in range(self.ROUNDS - 1):            new_l = (k[i] + self._ror(l[i], 8)) % self.MOD            new_l ^= i            new_k = self._rol(k[i], 3) ^ new_l            l.append(new_l)            k.append(new_k)        return k    def _ror(self, x, r):        return ((x >> r) | (x << (self.WORD_SIZE - r))) % self.MOD    def _rol(self, x, r):        return ((x << r) | (x >> (self.WORD_SIZE - r))) % self.MOD    def encrypt_block(self, block_bytes):        y, x = struct.unpack('>QQ', block_bytes)        for i in range(self.ROUNDS):            x = (self._ror(x, 8) + y) % self.MOD            x ^= self.key_schedule[i]            y = self._rol(y, 3) ^ x        return struct.pack('>QQ', y, x)    def decrypt_block(self, block_bytes):        y, x = struct.unpack('>QQ', block_bytes)        for i in range(self.ROUNDS - 1, -1, -1):            y = self._ror(y ^ x, 3)            x = self._rol((x ^ self.key_schedule[i]) - y % self.MOD, 8)        return struct.pack('>QQ', y, x)class SpeckEngine(CryptoEngine):    '''SPECK-128/128 in CTR mode + HMAC-SHA256 for AEAD'''    BLOCK_SIZE = 16    def encrypt(self, key, nonce, ad, plaintext):        cipher = SpeckCipher(key[:16])        # CTR mode encryption        ct = bytearray()        num_blocks = (len(plaintext) + self.BLOCK_SIZE - 1) // self.BLOCK_SIZE        for i in range(num_blocks):            counter = nonce[:8] + struct.pack('>Q', i)            keystream = cipher.encrypt_block(counter)            start = i * self.BLOCK_SIZE            block = plaintext[start:start + self.BLOCK_SIZE]            ct.extend(b ^ k for b, k in zip(block, keystream[:len(block)]))        # HMAC tag over ad + ct        mac_data = ad + bytes(ct)        tag = hmac.new(key, mac_data, hashlib.sha256).digest()[:16]        return bytes(ct), tag    def decrypt(self, key, nonce, ad, ciphertext, tag):        # Verify HMAC first        mac_data = ad + ciphertext        expected_tag = hmac.new(key, mac_data, hashlib.sha256).digest()[:16]        if not hmac.compare_digest(tag, expected_tag):            return None        # CTR mode decryption (same as encryption)        cipher = SpeckCipher(key[:16])        pt = bytearray()        num_blocks = (len(ciphertext) + self.BLOCK_SIZE - 1) // self.BLOCK_SIZE        for i in range(num_blocks):            counter = nonce[:8] + struct.pack('>Q', i)            keystream = cipher.encrypt_block(counter)            start = i * self.BLOCK_SIZE            block = ciphertext[start:start + self.BLOCK_SIZE]            pt.extend(b ^ k for b, k in zip(block, keystream[:len(block)]))        return bytes(pt)    def name(self): return 'SPECK-128/128-CTR-HMAC'    @property    def algo_id(self): return 32.4 PRESENT-80 Engine (Ultra-Lightweight Comparison)Create src/crypto/present_engine.py. This handles converting between our secure packet format and raw bytes:import structclass SecurePacket:    VERSION = 0x01    def __init__(self, algo_id, nonce, seq_num, timestamp, ad, ciphertext, tag):        self.algo_id = algo_id        self.nonce = nonce        # 64-bit bitmap    def check_and_update(self, seq_num: int) -> bool:        '''Returns True if packet is fresh, False if replay.'''        if seq_num == 0:            return False        if seq_num > self.highest_seq:            shift = seq_num - self.highest_seq            if shift >= self.window_size:                self.bitmap = 1            else:                self.bitmap = (self.bitmap << shift) | 1            self.highest_seq = seq_num            return True        diff = self.highest_seq - seq_num        if diff >= self.window_size:            return False  # Too old        bit = 1 << diff        if self.bitmap & bit:            return False  # Already seen (replay!)        self.bitmap |= bit        return True3.3 Sender NodeCreate src/network/sender.py:import socket, struct, os, timefrom src.network.packet import SecurePacketclass SecureSender:    def __init__(self, host, port, engine, key):        self.host = host        self.port = port        self.engine = engine        self.key = key        self.seq_num = 0    def send(self, message: str, device_id: str = 'SENDER01'):        self.seq_num += 1        nonce = struct.pack('>Q', self.seq_num) + os.urandom(8)        timestamp = int(time.time() * 1000)        ad = device_id.encode() + struct.pack('>Q', timestamp)        plaintext = message.encode('utf-8')        ct, tag = self.engine.encrypt(self.key, nonce, ad, plaintext)        pkt = SecurePacket(self.engine.algo_id, nonce, self.seq_num, timestamp, ad, ct, tag)        raw = pkt.serialize()        with socket.socket(socket.AF_INET, socket.SOCK_STREAM) as s:            s.connect((self.host, self.port))            s.sendall(struct.pack('>I', len(raw)) + raw)  # Literature Review & Background3.1 Evolution of Lightweight CryptographyResearch in lightweight cryptography began gaining momentum around 2007 with the publication of PRESENT, an ultra-lightweight block cipher designed for hardware-constrained environments. PRESENT operates on 64-bit blocks with 80-bit or 128-bit keys and uses a substitution-permutation network (SPN) structure with a 4-bit S-box applied in parallel. It requires only approximately 1,570 GE in its most compact hardware implementation, making it one of the smallest cipher designs. However, its 64-bit block size is now considered too small for modern security requirements due to birthday-bound concerns when encrypting large volumes of data.The NSA introduced the SIMON and SPECK cipher families in 2013. SIMON was optimized for hardware implementations using a balanced Feistel network with bitwise AND, XOR, and circular shift operations. SPECK was optimized for software performance using an add-rotate-XOR (ARX) structure. Both families offer flexibility with block sizes ranging from 32 to 128 bits and key sizes from 64 to 256 bits. SPECK-128/128, in particular, demonstrates throughput exceeding 3.5 Gbps in optimized ASIC implementations while maintaining a compact footprint.The NIST Lightweight Cryptography Standardization Process, launched in 2018, received 57 candidate submissions. Over three evaluation rounds spanning five years, these were narrowed to 10 finalists: ASCON, Elephant, GIFT-COFB, Grain-128AEAD, ISAP, Photon-Beetle, Romulus, Sparkle, TinyJambu, and Xoodyak. In February 2023, NIST selected ASCON as the winner, citing its strong security analysis, good all-around performance across hardware and software platforms, and the flexibility of its sponge-based design. The final standard, NIST SP 800-232, was published on August 13, 2025.3.2 The ASCON Cipher Family (NIST SP 800-232)ASCON was developed in 2014 by Christoph Dobraunig, Maria Eichlseder, Florian Mendel, and Martin Schlaffer from Graz University of Technology, Infineon Technologies, Lamarr Security Research, and Radboud University. It is built on a sponge construction (specifically, a MonkeyDuplex variant) that uses a single 320-bit permutation as the core primitive. This design means the same permutation can provide authenticated encryption (AEAD), hashing (Hash256), extendable output (XOF128), message authentication (MAC), and pseudo-random functions (PRF).The ASCON permutation operates on five 64-bit words (x0 through x4), forming a 320-bit state (S = x0 || x1 || x2 || x3 || x4). Each round of the permutation consists of three layers: (1) Addition of a round constant, (2) A substitution layer using a 5-bit S-box applied in a bitsliced fashion 64 times in parallel (the S-box is an affine equivalent of the Keccak chi function), and (3) A linear diffusion layer using rotations within each 64-bit word and XOR operations.VariantFunctionKey (bits)Nonce (bits)Tag (bits)Rate (bits)Rounds (a/b)Ascon-AEAD128Authenticated Encryption12812812812812/8Ascon-Hash256Hash Function---6412/12Ascon-XOF128Extendable Output---6412/12Ascon-CXOF128Custom XOF---6412/12Ascon-MacMessage Auth Code128-12825612/8Ascon-PrfPseudo-Random Fn128-Variable25612/83.3 ASCON-AEAD128 Encryption Process (Detailed)The authenticated encryption process has four distinct phases:Phase 1 - Initialization: The 320-bit state is initialized with an Initialization Vector (IV) that encodes the algorithm parameters, concatenated with the 128-bit secret key K and 128-bit nonce N. The strong permutation p[12] (12 rounds) is then applied, followed by XOR-ing the key into the capacity portion of the state. This ensures the initial state is thoroughly mixed and key-dependent.Phase 2 - Associated Data Processing: Associated data (AD) such as packet headers, device IDs, or metadata is divided into 128-bit blocks. Each block is XOR-ed into the rate portion of the state, followed by application of the reduced permutation p[8] (8 rounds). A domain separation constant is XOR-ed into the state after the last AD block. AD is authenticated but not encrypted, meaning it remains in plaintext but is covered by the authentication tag.Phase 3 - Plaintext Processing: Plaintext message blocks (128 bits each) are XOR-ed into the rate portion. The ciphertext is extracted from the rate bits before the permutation is applied. This duplex mode of operation means each plaintext block produces a ciphertext block incrementally, enabling streaming encryption without buffering the entire message.Phase 4 - Finalization: The key is XOR-ed into the state, the strong permutation p[12] is applied, and the key is XOR-ed again. The 128-bit authentication tag T is extracted from the last 128 bits of the state. During decryption, the computed tag is compared with the received tag; if they differ, the plaintext is discarded and an authentication failure is reported.3.4 Comparison of Lightweight AlgorithmsAlgorithmStructureBlock (bits)Key (bits)Gate EquivalentsBest ForStandardASCONSponge/SPNVariable (rate)128~2,570All-round IoTNIST SP 800-232AES-128SPN128128~2,400 (min)General purposeFIPS 197SPECK-128/128ARX Feistel128128~1,280Software-first IoTISO 29167-22PRESENT-80SPN6480~1,570Ultra-compact HWISO 29192-2SIMON-128/128Feistel128128~1,234Hardware-first IoTISO 29167-21GIFT-COFBSPN128128~2,900AEAD for IoTNIST LWC finalist4. System Architecture & Design4.1 High-Level Architecture OverviewThe system follows a three-tier architecture: (1) Device Layer consisting of sender and receiver nodes, (2) Communication Layer handling wireless packet transmission, and (3) Security Layer providing cryptographic services. Each tier is modular, allowing algorithm substitution for benchmarking.4.2 System ComponentsSender Node: Captures or generates sensor data (temperature, humidity, GPS coordinates, or text messages), serializes the payload, invokes the cryptographic engine to produce authenticated ciphertext, constructs a secure packet with header (nonce, device ID, sequence number, timestamp) and encrypted payload plus authentication tag, and transmits via Wi-Fi socket.Receiver Node: Listens on a designated port, receives the secure packet, extracts the header fields, invokes the cryptographic engine for decryption and tag verification, checks the sequence number against a replay-detection window, and either accepts the plaintext or rejects the packet on authentication failure.Key Management Module: For this project, a pre-shared key (PSK) model is used. Keys are generated using a cryptographically secure random number generator (os.urandom in Python, or hardware RNG on ESP32), stored in encrypted flash storage on the device, and rotated at configurable intervals. A Diffie-Hellman key exchange over the secure channel can be implemented as an extension.Benchmarking Engine: A dedicated module that runs each algorithm (ASCON, AES-128, SPECK, PRESENT) on identical inputs and measures execution time, memory usage, throughput, and energy proxy (CPU cycles). Results are exported as CSV for analysis.4.3 Secure Packet FormatFieldSize (bytes)DescriptionVersion1Protocol version (0x01)Algorithm ID10x01=ASCON, 0x02=AES-GCM, 0x03=SPECK, 0x04=PRESENTNonce16Unique per message; generated from counter + random saltSequence Number4Monotonically increasing; used for replay detectionTimestamp8Unix timestamp in milliseconds; for freshness checkAD Length2Length of associated data that followsAssociated DataVariablePlaintext metadata (device ID, sensor type, etc.)Ciphertext Length2Length of encrypted payloadCiphertextVariableEncrypted payload dataAuth Tag16128-bit authentication tag from AEAD4.4 Communication Flow (Sequence)Step 1 - Key Pre-loading: Before deployment, both sender and receiver are provisioned with a shared 128-bit secret key K. In a lab environment, this can be done via USB serial; in production, a secure provisioning protocol like ECDH would be used.Step 2 - Sender Encrypts: Sender generates a fresh 128-bit nonce N (using a counter concatenated with 64 random bits to prevent nonce reuse), constructs the associated data (device ID + timestamp), calls ascon_encrypt(K, N, AD, plaintext) to get ciphertext C and tag T, builds the packet, and sends it over a TCP/UDP socket.Step 3 - Receiver Decrypts: Receiver extracts N and AD from the packet header, calls ascon_decrypt(K, N, AD, C||T). If the tag verification passes, the plaintext is accepted. If it fails (indicating tampering, corruption, or wrong key), the packet is silently dropped and the event is logged.Step 4 - Replay Check: Receiver maintains a sliding window of accepted sequence numbers (window size W=64). Any packet with a sequence number below the window lower bound or already in the window bitmap is rejected.4.5 Replay Detection AlgorithmThe anti-replay mechanism uses a sliding window approach inspired by IPsec ESP (RFC 4303). A 64-bit bitmap tracks which of the last 64 sequence numbers have been received. The algorithm proceeds as follows: If the received sequence number is less than (highest_seq - window_size), reject. If it is greater than highest_seq, advance the window, mark as received, and accept. If it falls within the window, check the corresponding bit in the bitmap; if already set, reject as duplicate; otherwise mark and accept.5. AES-128-GCM will be competitive on hardware with AES-NI instructions but slower on platforms without. ASCON will be moderately fast, slightly behind SPECK and AES on high-end CPUs, but will excel in combined AEAD+hash scenarios due to its unified design. PRESENT will be the slowest in software due to its hardware-oriented 4-bit S-box design.On constrained 8-bit microcontrollers, SPECK and ASCON typically outperform AES significantly. On 32-bit ARM (like ESP32), ASCON achieves competitive throughput. The key insight for your project analysis is that ASCON provides the best security-to-cost ratio when both encryption and hashing are needed, which is the typical IoT scenario.8. Attack Simulation & Security Analysis8.1 Eavesdropping Attack DemonstrationSimulate a passive eavesdropper using Scapy or raw socket sniffing. The attacker captures packets in transit but cannot decrypt them:Setup: Run the sender and receiver on different terminals. Run a third terminal with Scapy's sniff() function to capture packets. Show that captured packets contain only ciphertext (random-looking bytes) with no plaintext leakage. Display the hex dump of captured data alongside the actual plaintext to visually demonstrate the encryption.Defense Proof: ASCON-AEAD128 provides IND-CPA (indistinguishability under chosen-plaintext attack) security. The ciphertext is computationally indistinguishable from random bytes to any adversary who does not possess the 128-bit key.8.2 Replay Attack DemonstrationSetup: The attacker captures a valid encrypted packet and replays it (sends the exact same bytes again). First, demonstrate the attack succeeding against a system WITHOUT replay protection. Then enable the sliding-window replay guard and show the replayed packet being rejected with a REPLAY_DETECTED log entry.Defense Proof: The sliding window tracks sequence numbers. Since the nonce includes the sequence number, even if the attacker replays the packet, the receiver's replay guard rejects it based on the duplicate sequence number. Additionally, the timestamp in the associated data can be used for a freshness check (reject packets older than 30 seconds).8.3 Man-in-the-Middle Attack DemonstrationSetup: The attacker intercepts a packet, modifies the ciphertext (flips some bits), and forwards the modified packet. Alternatively, the attacker replaces the entire ciphertext with their own fabricated ciphertext.Defense Proof: ASCON's 128-bit authentication tag provides INT-CTXT (integrity of ciphertext) security. Any modification to the ciphertext, associated data, or nonce will cause tag verification to fail during decryption. The probability of a random modification passing the tag check is 1 in 2 to the power of 128, which is negligible.8.4 Security Properties SummaryPropertyMechanismASCON GuaranteeConfidentialitySponge-based encryption with 128-bit key128-bit security (IND-CPA)Integrity128-bit authentication tag128-bit security (INT-CTXT)AuthenticityAEAD binds key to both plaintext and ADOnly key holder can produce valid tagsReplay ProtectionNonce + sequence number + sliding windowDuplicate packets rejectedFreshnessTimestamp in associated dataStale packets rejected (configurable window)Nonce Misuse ResilienceCounter-based nonce generationGuaranteed unique per key/nonce pair9.
