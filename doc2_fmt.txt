Literature Review & Background`n3.1 Evolution of Lightweight CryptographyResearch in lightweight cryptography began gaining momentum around 2007 with the publication of PRESENT, an ultra-lightweight block cipher designed for hardware-constrained environments. PRESENT operates on 64-bit blocks with 80-bit or 128-bit keys and uses a substitution-permutation network (SPN) structure with a 4-bit S-box applied in parallel. It requires only approximately 1,570 GE in its most compact hardware implementation, making it one of the smallest cipher designs. However, its 64-bit block size is now considered too small for modern security requirements due to birthday-bound concerns when encrypting large volumes of data.The NSA introduced the SIMON and SPECK cipher families in `n2`n0`n1`n3. SIMON was optimized for hardware implementations using a balanced Feistel network with bitwise AND, XOR, and circular shift operations. SPECK was optimized for software performance using an add-rotate-XOR (ARX) structure. Both families offer flexibility with block sizes ranging from 32 to 128 bits and key sizes from 64 to 256 bits. SPECK-128/128, in particular, demonstrates throughput exceeding `n3.5 Gbps in optimized ASIC implementations while maintaining a compact footprint.The NIST Lightweight Cryptography Standardization Process, launched in 2018, received 57 candidate submissions. Over three evaluation rounds spanning five years, these were narrowed to 10 finalists: ASCON, Elephant, GIFT-COFB, Grain-128AEAD, ISAP, Photon-Beetle, Romulus, Sparkle, TinyJambu, and Xoodyak. In February 2023, NIST selected ASCON as the winner, citing its strong security analysis, good all-around performance across hardware and software platforms, and the flexibility of its sponge-based design. The final standard, NIST SP 800-232, was published on August 13, 2025.`n3.2 The ASCON Cipher Family (NIST SP 800-232)ASCON was developed in 2014 by Christoph Dobraunig, Maria Eichlseder, Florian Mendel, and Martin Schlaffer from Graz University of Technology, Infineon Technologies, Lamarr Security Research, and Radboud University. It is built on a sponge construction (specifically, a MonkeyDuplex variant) that uses a single 320-bit permutation as the core primitive. This design means the same permutation can provide authenticated encryption (AEAD), hashing (Hash256), extendable output (XOF128), message authentication (MAC), and pseudo-random functions (PRF).The ASCON permutation operates on five 64-bit words (x0 through x4), forming a 320-bit state (S = x0 || x1 || x2 || x3 || x4). Each round of the permutation consists of three layers: (1) Addition of a round constant, (2) A substitution layer using a 5-bit S-box applied in a bitsliced fashion 64 times in parallel (the S-box is an affine equivalent of the Keccak chi function), and (3) A linear diffusion layer using rotations within each 64-bit word and XOR operations.VariantFunctionKey (bits)Nonce (bits)Tag (bits)Rate (bits)Rounds (a/b)Ascon-AEAD128Authenticated Encryption12812812812812/8Ascon-Hash256Hash Function---6412/12Ascon-XOF128Extendable Output---6412/12Ascon-CXOF128Custom XOF---6412/12Ascon-MacMessage Auth Code128-12825612/8Ascon-PrfPseudo-Random Fn128-Variable25612/`n8`n3.3 ASCON-AEAD128 Encryption Process (Detailed)The authenticated encryption process has four distinct phases:Phase 1 - Initialization: The 320-bit state is initialized with an Initialization Vector (IV) that encodes the algorithm parameters, concatenated with the 128-bit secret key K and 128-bit nonce N. The strong permutation p[12] (12 rounds) is then applied, followed by XOR-ing the key into the capacity portion of the state. This ensures the initial state is thoroughly mixed and key-dependent.Phase 2 - Associated Data Processing: Associated data (AD) such as packet headers, device IDs, or metadata is divided into 128-bit blocks. Each block is XOR-ed into the rate portion of the state, followed by application of the reduced permutation p[8] (8 rounds). A domain separation constant is XOR-ed into the state after the last AD block. AD is authenticated but not encrypted, meaning it remains in plaintext but is covered by the authentication tag.Phase 3 - Plaintext Processing: Plaintext message blocks (128 bits each) are XOR-ed into the rate portion. The ciphertext is extracted from the rate bits before the permutation is applied. This duplex mode of operation means each plaintext block produces a ciphertext block incrementally, enabling streaming encryption without buffering the entire message.Phase 4 - Finalization: The key is XOR-ed into the state, the strong permutation p[12] is applied, and the key is XOR-ed again. The 128-bit authentication tag T is extracted from the last 128 bits of the state. During decryption, the computed tag is compared with the received tag; if they differ, the plaintext is discarded and an authentication failure is reported.`n3.4 Comparison of Lightweight AlgorithmsAlgorithmStructureBlock (bits)Key (bits)Gate EquivalentsBest ForStandardASCONSponge/SPNVariable (rate)128~2,570All-round IoTNIST SP 800-232AES-128SPN128128~2,400 (min)General purposeFIPS 197SPECK-128/128ARX Feistel128128~1,280Software-first IoTISO 29167-22PRESENT-80SPN6480~1,570Ultra-compact HWISO 29192-2SIMON-128/128Feistel128128~1,234Hardware-first IoTISO 29167-21GIFT-COFBSPN128128~2,900AEAD for IoTNIST LWC finalist`n4. System Architecture & Design`n4.1 High-Level Architecture OverviewThe system follows a three-tier architecture: (1) Device Layer consisting of sender and receiver nodes, (2) Communication Layer handling wireless packet transmission, and (3) Security Layer providing cryptographic services. Each tier is modular, allowing algorithm substitution for benchmarking.`n4.2 System ComponentsSender Node: Captures or generates sensor data (temperature, humidity, GPS coordinates, or text messages), serializes the payload, invokes the cryptographic engine to produce authenticated ciphertext, constructs a secure packet with header (nonce, device ID, sequence number, timestamp) and encrypted payload plus authentication tag, and transmits via Wi-Fi socket.Receiver Node: Listens on a designated port, receives the secure packet, extracts the header fields, invokes the cryptographic engine for decryption and tag verification, checks the sequence number against a replay-detection window, and either accepts the plaintext or rejects the packet on authentication failure.Key Management Module: For this project, a pre-shared key (PSK) model is used. Keys are generated using a cryptographically secure random number generator (os.urandom in Python, or hardware RNG on ESP32), stored in encrypted flash storage on the device, and rotated at configurable intervals. A Diffie-Hellman key exchange over the secure channel can be implemented as an extension.Benchmarking Engine: A dedicated module that runs each algorithm (ASCON, AES-128, SPECK, PRESENT) on identical inputs and measures execution time, memory usage, throughput, and energy proxy (CPU cycles). Results are exported as CSV for analysis.`n4.3 Secure Packet FormatFieldSize (bytes)DescriptionVersion1Protocol version (0x01)Algorithm ID10x01=ASCON, 0x02=AES-GCM, 0x03=SPECK, 0x04=PRESENTNonce16Unique per message; generated from counter + random saltSequence Number4Monotonically increasing; used for replay detectionTimestamp8Unix timestamp in milliseconds; for freshness checkAD Length2Length of associated data that followsAssociated DataVariablePlaintext metadata (device ID, sensor type, etc.)Ciphertext Length2Length of encrypted payloadCiphertextVariableEncrypted payload dataAuth Tag16128-bit authentication tag from AEAD`n4.4 Communication Flow (Sequence)Step 1 - Key Pre-loading: Before deployment, both sender and receiver are provisioned with a shared 128-bit secret key K. In a lab environment, this can be done via USB serial; in production, a secure provisioning protocol like ECDH would be used.Step 2 - Sender Encrypts: Sender generates a fresh 128-bit nonce N (using a counter concatenated with 64 random bits to prevent nonce reuse), constructs the associated data (device ID + timestamp), calls ascon_encrypt(K, N, AD, plaintext) to get ciphertext C and tag T, builds the packet, and sends it over a TCP/UDP socket.Step 3 - Receiver Decrypts: Receiver extracts N and AD from the packet header, calls ascon_decrypt(K, N, AD, C||T). If the tag verification passes, the plaintext is accepted. If it fails (indicating tampering, corruption, or wrong key), the packet is silently dropped and the event is logged.Step 4 - Replay Check: Receiver maintains a sliding window of accepted sequence numbers (window size W=64). Any packet with a sequence number below the window lower bound or already in the window bitmap is rejected.`n4.5 Replay Detection AlgorithmThe anti-replay mechanism uses a sliding window approach inspired by IPsec ESP (RFC 4303). A 64-bit bitmap tracks which of the last 64 sequence numbers have been received. The algorithm proceeds as follows: If the received sequence number is less than (highest_seq - window_size), reject. If it is greater than highest_seq, advance the window, mark as received, and accept. If it falls within the window, check the corresponding bit in the bitmap; if already set, reject as duplicate; otherwise mark and accept.`n5. AES-128-GCM will be competitive on hardware with AES-NI instructions but slower on platforms without. ASCON will be moderately fast, slightly behind SPECK and AES on high-end CPUs, but will excel in combined AEAD+hash scenarios due to its unified design. PRESENT will be the slowest in software due to its hardware-oriented 4-bit S-box design.On constrained 8-bit microcontrollers, SPECK and ASCON typically outperform AES significantly. On 32-bit ARM (like ESP32), ASCON achieves competitive throughput. The key insight for your project analysis is that ASCON provides the best security-to-cost ratio when both encryption and hashing are needed, which is the typical IoT scenario.`n8. Attack Simulation & Security Analysis`n8.1 Eavesdropping Attack DemonstrationSimulate a passive eavesdropper using Scapy or raw socket sniffing. The attacker captures packets in transit but cannot decrypt them:Setup: Run the sender and receiver on different terminals. Run a third terminal with Scapy's sniff() function to capture packets. Show that captured packets contain only ciphertext (random-looking bytes) with no plaintext leakage. Display the hex dump of captured data alongside the actual plaintext to visually demonstrate the encryption.Defense Proof: ASCON-AEAD128 provides IND-CPA (indistinguishability under chosen-plaintext attack) security. The ciphertext is computationally indistinguishable from random bytes to any adversary who does not possess the 128-bit key.`n8.2 Replay Attack DemonstrationSetup: The attacker captures a valid encrypted packet and replays it (sends the exact same bytes again). First, demonstrate the attack succeeding against a system WITHOUT replay protection. Then enable the sliding-window replay guard and show the replayed packet being rejected with a REPLAY_DETECTED log entry.Defense Proof: The sliding window tracks sequence numbers. Since the nonce includes the sequence number, even if the attacker replays the packet, the receiver's replay guard rejects it based on the duplicate sequence number. Additionally, the timestamp in the associated data can be used for a freshness check (reject packets older than 30 seconds).`n8.3 Man-in-the-Middle Attack DemonstrationSetup: The attacker intercepts a packet, modifies the ciphertext (flips some bits), and forwards the modified packet. Alternatively, the attacker replaces the entire ciphertext with their own fabricated ciphertext.Defense Proof: ASCON's 128-bit authentication tag provides INT-CTXT (integrity of ciphertext) security. Any modification to the ciphertext, associated data, or nonce will cause tag verification to fail during decryption. The probability of a random modification passing the tag check is 1 in 2 to the power of 128, which is negligible.`n8.4 Security Properties SummaryPropertyMechanismASCON GuaranteeConfidentialitySponge-based encryption with 128-bit key128-bit security (IND-CPA)Integrity128-bit authentication tag128-bit security (INT-CTXT)AuthenticityAEAD binds key to both plaintext and ADOnly key holder can produce valid tagsReplay ProtectionNonce + sequence number + sliding windowDuplicate packets rejectedFreshnessTimestamp in associated dataStale packets rejected (configurable window)Nonce Misuse ResilienceCounter-based nonce generationGuaranteed unique per key/nonce pair`n9.
